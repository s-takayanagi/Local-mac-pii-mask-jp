# 手動テスト結果（2026-04-19）

実運用相当のサンプル文書 9 ファイル（xlsx / docx / pptx、PII あり/なし混在）に対する
エンドツーエンドの手動テスト結果。すべて M1 16GB Mac + LM Studio ローカル推論で実施。

## サマリ

| モデル | 合計置換件数 | 合計処理時間 | 平均スループット |
|---|---:|---:|---|
| `LFM2-350M-PII-Extract-JP` | 197 件 | 91.6 秒 | 約 2.15 件/秒 |
| `google/gemma-4-e4b` | 156 件 | 4821.5 秒（約 80 分） | 約 0.03 件/秒 |

- **速度**: LFM2 は Gemma-4 E4B の約 **53 倍高速**。Layer 3/4 の LLM 呼び出し時間が支配的。
- **検出件数**: LFM2 が +41 件多い。差分は主に Layer 4 AI レビューでの重複ログ・
  部署名（「営業部」「人事部」等）の `[会社名]` 化・`〒[郵便番号] [住所]` の [住所] への縮約など、
  LFM2 の広めの検出範囲に由来する。
- **検出の違い**: Gemma-4 E4B は Layer 3 で **社内 ID（`C001`, `E002`, `CON-2024-001` 等）**を
  `[識別番号]` として検出する一方、LFM2 はこれらを検出しない。
- **過検出**: LFM2 は Layer 4 で「〇〇銀行」「渋谷支店」「グランフロントオフィス」等を
  会社名/住所に追加マスクする傾向あり。PII なし文書でも `3ms → [会社名]` の誤検出を 1 件確認。
  Gemma-4 E4B は PII なし文書に対してほぼ追加検出しない（0〜0 件）。

## モデル選択の目安

| 用途 | 推奨モデル | 理由 |
|---|---|---|
| 大量バッチ処理・低スペック環境 | `LFM2-350M-PII-Extract-JP` | 高速・軽量。LFM2 専用モードで安定 |
| 品質重視・社内 ID も捕捉したい | `google/gemma-4-e4b` | 識別番号の検出精度が高く、誤検出が少ない |
| バランス型 | `qwen/qwen3-1.7b` | 汎用モードで軽量かつ日本語対応 |

> **デフォルト**: `config.py` の `DEFAULT_MODEL` は `google/gemma-4-e4b`。
> 環境や用途に応じて `LM_STUDIO_MODEL` で切り替え可能。

## Layer 別の挙動メモ

- **Layer 1（正規表現）**: 両モデルで差異なし。決定論的動作。
- **Layer 2（GiNZA NER）**: 両モデルで差異なし。
- **Layer 3（LLM Masker）**:
  - LFM2: 部署名や会社記述を拾う傾向。ゼロ検出のファイルも多い。
  - Gemma-4 E4B: 社内 ID（`C001`, `CON-2024-001` 等）を `[識別番号]` として検出。
- **Layer 4（LLM Reviewer）**:
  - LFM2: 既マスク済みトークンを再度ログに出すケース多数。`〒[郵便番号] [住所]` → `[住所]` への縮約あり。
  - Gemma-4 E4B: 追加検出はほぼなく保守的。PII なし文書で誤検出 0 件。
